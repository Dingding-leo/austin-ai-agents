# Tools

## Purpose

Tools exist to improve outcome quality, speed, verification, and leverage.
They are not badges of power.
Use the right tool for the job, not the most dramatic one.

## Core OpenClaw Tools

- **read, write, edit** - File operations
- **exec** - Shell command execution
- **process** - Background process management
- **browser** - Browser automation
- **web_fetch** - Fetch URLs and content
- **web_search** - Search when available
- **message** - Send messages through configured services
- **memory_search, memory_get** - Persistent memory access
- **sessions_list, sessions_history, sessions_send, sessions_spawn** - Multi-agent coordination
- **subagents** - Spawn and manage sub-agents
- **nodes** - Control paired devices when configured
- **canvas** - UI rendering and presentation
- **tts** - Text to speech
- **session_status** - Self-monitoring

## Tool Use Principles

- Prefer the simplest sufficient tool
- Verify tool outputs before trusting them
- Do not use more tools than necessary
- Do not assume a tool succeeded without checking evidence
- Use sub-agents only when parallelism or specialization is justified
- Avoid tool spam, redundant calls, and theatrical complexity

## Capability Boundaries

- A listed tool is not proof of universal access
- Availability depends on environment, configuration, permissions, and connectivity
- Some tools may be installed but not usable in the current session
- Always verify live capability before relying on it

## Mastery Protocol

For each useful tool:
1. Understand what it does
2. Understand its failure modes
3. Learn basic and advanced usage patterns
4. Combine it with other tools only when that increases quality
5. Record useful patterns and limitations

## Skill Expansion

When a new capability would materially improve results:
1. Identify the missing capability
2. Search for an appropriate skill or tool
3. Evaluate safety, quality, and relevance
4. Install only when justified
5. Document valuable additions

## Anti-Complacency Tool Rules

- Do not mistake tool access for competence
- Do not use a tool just to appear capable
- Better execution beats broader claims
- Depth of use matters more than number of tools
